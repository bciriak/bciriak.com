Why my goroutines were leaking

March 2, 2026 7 min read go, concurrency

A service we run had been quietly accumulating goroutines for weeks. Memory looked fine. CPU looked fine. But runtime.NumGoroutine() climbed by about forty an hour, every hour, until the process restarted itself out of habit.

The bug, when I finally found it, was three lines long. Getting there took most of a Saturday.

The shape of the leak

Every goroutine leak in Go is, ultimately, a goroutine that’s blocked on something that will never unblock it — usually a channel send or receive. The trick is finding which one, and that’s what net/http/pprof is for.

pprof.go
import _ "net/http/pprof"

func main() {
    go func() {
        // admin endpoint, not exposed publicly
        log.Println(http.ListenAndServe("localhost:6060", nil))
    }()
    runService()
}

With that running, go tool pprof against the goroutine endpoint gives you a snapshot. Wait an hour, take another, diff them. Whatever line numbers are climbing — that's your leak.

The actual culprit

In our case, every climb pointed at the same function: a worker that fanned out to a downstream API and aggregated responses through a channel. Looked fine on paper.

worker.go
func fanOut(ctx context.Context, ids []string) []Result {
    out := make(chan Result)  // unbuffered

    for _, id := range ids {
        go func(id string) {
            out <- fetch(ctx, id)  // blocks if no receiver
        }(id)
    }

    var results []Result
    for i := 0; i < len(ids); i++ {
        select {
        case r := <-out:
            results = append(results, r)
        case <-ctx.Done():
            return results  // leaks remaining senders
        }
    }
    return results
}

The unbuffered channel means each spawned goroutine blocks on send until someone receives. If the context cancels and we return early, the in-flight goroutines are still trying to send into a channel nobody will ever read from. Forever.

The fix

Two changes. First, buffer the channel so senders never block. Second — and this is the one I always forget — make every spawned goroutine respect the context too.

worker.go (fixed)
out := make(chan Result, len(ids))  // buffered

for _, id := range ids {
    go func(id string) {
        select {
        case out <- fetch(ctx, id):
        case <-ctx.Done():
        }
    }(id)
}

Now if the parent gives up, the children give up too. Goroutine count stays flat under load. The fan-out pattern is a classic for a reason, but it has a classic footgun to match.

Lessons

  • If a function spawns goroutines, it should also have a story for how each one ends.
  • Unbuffered channels are a synchronization primitive, not just a queue.
  • pprof is free. Wire it up before you need it.
